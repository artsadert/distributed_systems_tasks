## Анализ и выводы

### Пропускная способность

- При размере 128 B и целевой интенсивности 10000 msg/s:
  - RabbitMQ фактически отправил **400 msg/s**
  - Redis фактически отправил **4919 msg/s**
  - artmq фактически отправил **437 msg/s**
- Redis показал на **1128.6%** более высокую пропускную способность на малых сообщениях.
- artmq близок по пропускной способности к RabbitMQ (оба ограничены синхронным циклом publish→ack
  у продюсера) и значительно отстаёт от Redis, который не подтверждает каждое сообщение.

### Влияние размера сообщения

- **128 B**: RabbitMQ p95 = 1.00 ms, Redis p95 = 0.00 ms, artmq p95 = 11.00 ms. Минимальная задержка у Redis.
- **1024 B**: RabbitMQ p95 = 1.00 ms, Redis p95 = 0.00 ms, artmq p95 = 2.00 ms. Минимальная задержка у Redis.
- **10240 B**: RabbitMQ p95 = 1.00 ms, Redis p95 = 0.00 ms, artmq p95 = 13.00 ms. Минимальная задержка у Redis.
- **102400 B**: RabbitMQ p95 = 1.00 ms, Redis p95 = 1.00 ms, artmq p95 = 15.00 ms. RabbitMQ и Redis сопоставимы; artmq заметно медленнее на крупных payload.

### Устойчивость к высокой нагрузке

Точка деградации определялась по увеличению p95 задержки > 50 мс или потерям > 1%.

**RabbitMQ**:

- Размер 128 B: стабильно работает до 10000 msg/s.
- Размер 1024 B: стабильно работает до 10000 msg/s.
- Размер 10240 B: стабильно работает до 10000 msg/s.
- Размер 102400 B: стабильно работает до 10000 msg/s.

**Redis**:

- Размер 128 B: стабильно работает до 10000 msg/s.
- Размер 1024 B: стабильно работает до 10000 msg/s.
- Размер 10240 B: стабильно работает до 10000 msg/s.
- Размер 102400 B: стабильно работает до 10000 msg/s.

**artmq**:

- Размер 128 B: потерь нет, p95 = 11–12 ms во всём диапазоне.
- Размер 1024 B: потерь нет, p95 ≈ 2 ms — наиболее «комфортный» размер.
- Размер 10240 B: потерь нет, но p95 уже 11–13 ms.
- Размер 102400 B: потерь нет, p95 растёт до 17 ms при 10000 msg/s.
- Деградация по критерию p95 > 50 мс не достигнута, но сам уровень latency
  заметно выше, чем у RabbitMQ и Redis из-за синхронного PUBACK и self-hosted
  реализации без оптимизаций сетевого I/O.

### Рекомендации

- **Redis** предпочтителен, когда:
  - Сообщения маленькие (до 10 КБ) и требуется минимальная задержка.
  - Высокая пропускная способность критичнее гарантий доставки.
- **RabbitMQ** следует выбрать, если:
  - Важна надёжность: сообщения не должны теряться.
  - Размер сообщений большой (более 10 КБ) или нагрузка неравномерная.
- **artmq** имеет смысл рассматривать, когда:
  - Нужен встраиваемый MQTT 5.0 брокер «своими силами» с QoS 1/2 и приоритетами/TTL.
  - Допустима latency 2–17 ms и пропускная способность сотен msg/s — типично для IoT
    и устройств, а не для высокочастотного backend-обмена.
- Для сценариев с жёсткими требованиями к latency на малых payload лучше Redis.
- Для систем, где потеря данных недопустима — RabbitMQ или artmq (QoS 1/2).
